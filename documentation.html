Lorg in a nutshell

Lorg defines a syntax to organize hierarchical data. Hierarchical data are components containing other components. Here is a full example of a Lorg document used to prepare the gear for a trek. 

# Sleep
    ## Shelter
        Should I use a tent instead?
        ### Trekking poles
            $ Cost: 69
            $ Weight: 560
        ### Tarp
            $ Cost: 60
            $ Weight: 495
    ## Sleeping bag
        $ Cost: 51.99
        $ Weight: 1200.00
        ### Liner
            $ Cost: 50
            $ Weight: 340
# Clothes
    I should bring a jacket.
    And maybe a beanie.
    ## Poncho
        $ Cost: 49
        $ Weight: 400
    ## Tee-shirt
        $ Cost: 0
        $ Weight: 150
    ## Shoes
        $ Cost: 169
        $ Weight: 0
        I wear them so no weight in the backpack.

Your goal with this document is to have the total Cost and Weight of your gear.

By glancing at the example you can certainly already create your own Lorg documents, but let's explain how this document is structured.

Leading and trailing spaces

Leading spaces (spaces at the beginning of a line) and trailing spaces (spaces at the end of a line) are ignored. They can be used to make the visualisation of the hierarchy easier.

Nodes

In Lorg, components are call nodes. Nodes have a title, children and units.

In this example, there is a node with the title Poncho. It is the child of the node with the title Clothes. It defines two units: Cost and Weight.

If you know Markdown, you already know how to structure your nodes. You use # to define the hierarchy.

Multiple nodes can have the same title.

In our example:

  • The node with a title Clothes has a heading level of 1.
  • The node with a title Sleep is a sibling of Clothes.
  • The node with a title Poncho is a child of Clothes.

Units

In Lorg, we define units. They have a name and a value. We associate nodes with units.

To define a unit, start a line with a $ followed by the unit name then : then the unit value.

A unit name can contain spaces and :. The names Weight (g) and expenses:groceries are valid.

A unit value can be an integer or a floating number. The number must be in decimal (example: 34.5). Hexadecimal values (example: 0xFF) are invalid.

If you do not know the value of a unit in a node: do not define it! The values will be calculated by Lorg according the unit values in the node children. This is the whole point of Lorg!

For the moment, Lorg only sum the values. Maybe in the future other operations will be available (average, min, max...).

Be careful: you cannot define multiple time in the same node a unit.

In our example:

  • Tee-shirt has a Cost of 0 and a Weight of 150.
  • Clothes does not have any units. Lorg will calculate for it its associated Cost and Weight units.

Comments

All lines that are not node definitions or unit definitions are comments Comments are ignored by Lorg when parsing a document.

In our example, the line I should bring a jacket. is ignored.

Lorg gives the total for each unit.

When a unit was not defined in the document, Lorg creates it. If the node does not have children, the value is 0. Else the value is calculated according to the children. In all cases, those created units are indicated by [Calculated]

A unit defined in the document can be ignored. This means that its value is not used for the total. It happens when a parent already set a value. Those ignored units are indicated by [Ignored]

In our example:

  • Clothes now has units. The units values are the sum of the unit values of its children.
  • The unit values of Liner are ignored because its parent Sleeping bag has already defined values for the same units.

Using Lorg

Lorg is available as a command line interface or as a web app.

You can download the CLI version. When using the CLI version, put your Lorg in a file (for example here trek.lorg) and use the lorg command. 

lorg trek.lorg

Or you can use the web app.

Lorg is open source and you can access to the source code

You now have all the basis to use Lorg. We hope Lorg will be useful for you!

Syntax definition in English

Here is the syntax definition written in English. Lorg syntax was designed to be easy to pick up so you can certainly use Lorg without reading this section.

Various

Any character is any character except the new line character.

A white space can be the space character or the tabulation character.

Optional white spaces is zero or multiple white spaces.

Leading spaces is optional white spaces at the beginning of a line.

Trailing spaces is optional white spaces at the end of a line.

A digit is 0 or 1 or 2 or 3 or 4 or 5 or 6 or 7 or 8 or 9.

An integral is at least one digit.

A float is at least one digit then . then at least one digit.

A number is a float or an integral. float definition has a precedence over integral definition.

Nodes

A node heading contains one or more #.

A node title contains non white space then zero or more any character, and must end with a non white space.

A node definition is defined in a full line from the first character of the line to the last character.

A node definition ignores leading spaces and trailing spaces.

A node definition contains a node heading then optional white spaces then a node title.

Then number of # in the node heading defines the node level.

In a Lorg document, there can be multiple node definitions with a node level of 1. For nodes with a superior node level, the node level must not be superior to 1 + the node level of the previous node definition. In other words, a node must be the sibling or the direct descendant of another node.

If a line starts a node definition but it is invalid (for example it does not contain the node title), then this is an error.

Units

A unit name contains non white space then zero or more any character, and must end with a non white space.

A unit value is a number.

A unit definition is defined in a full line from the first character of the line to the last characters.

A node definition ignores leading spaces and trailing spaces.

A unit definition contains a $ then optional white spaces then a unit name then optional white spaces then a : then optional white spaces then a unit value.

In the unit definition, this means that unit names can contain one or more : and white spaces.

A unit definition must follow, but not necessarily directly, a node definition. In other words, a unit definition must be associated with a node definition.

If a line starts a unit definition but it is invalid (for example it does not contain a unit value), then this is an error.

Comments

All lines that are not node definitions or unit definitions are comments; they are ignored.

Syntax definition in regular expressions

Here is the syntax definition in JavaScript (at least with the ECMAScript 6).

Note that of course you would not implement those exact regular expressions if you use regular expressions in you own Lorg implementation. The purpose here is to explicitly show the definitions of all the terms. For example to catch a node definition in a line in your implementation, you could trim the white spaces then use a simple regular expression like ^(#+)\s*(.+)$. 

// We use ?: for not capturing the unnecessary groups.

any_character = `[^\n]`
white_space = `[ \t]`
non_white_space = `[^ \t]`
optional_white_space = `${white_space}*`
leading_white_space = `^${white_space}*`
trailing_white_space = `${white_space}*$`

sign = `[-+]`
digit = `[0-9]`
integral = `${sign}?${digit}+`
float = `${sign}?${digit}+\\.${digit}+`
// This is important for numbers to catch floats before integrals.
number = `(?:${float}|${integral})`

node_heading = `#+`
node_title = `(?:${non_white_space}${any_character}*${non_white_space}|${non_white_space})`
node_definition = `${leading_white_space}(${node_heading})${optional_white_space}(${node_title})${trailing_white_space}`

unit_name = `(?:${non_white_space}${any_character}*${non_white_space}|${non_white_space})`
unit_value = number
unit_definition = `${leading_white_space}\\$${optional_white_space}(${unit_name})${optional_white_space}:${optional_white_space}(${unit_value})${trailing_white_space}`

At the end of this page, there is a full JavaScript code to test those regular expressions. You can use those tests to test your own implementation.

Behaviour

When Lorg parses a Lorg document, it creates a tree of nodes. A node contains a title, other nodes (its children) and units.

Multiple nodes can have the same title.

A unit value can be considered as real or calculated. A real unit value is a value explicitly defined in the Lorg document. A calculated unit value is a value calculated by Lorg.

When a unit is not defined in the Lorg document for a node but defined for another node, Lorg creates this unit and set it as calculated. If the node does not have children then the unit value is set to the default unit value (which is0); if the node has children then the unit value is the sum of the children unit values.

A unit can be defined only once in a node.

A unit can be considered as ignored. When a unit value is considered as real in a node, then all the unit values of the children of this node for this unit are considered ignored. This means that those ignored unit values are not used to get the unit value for the highest node parent (the node of node level 1).

Lorg automatically creates a main node called the total node. All nodes with a node level of 1 are the children of this total node. In the total node calculates its unit values as it is done for any other nodes. The Lorg software should contain an option for hiding the total node and an option for changing its title.

Tests in JavaScript

If you want to test your implementation in JavaScript, here are some tests. 

function test_is_full_match(string, regex_name, regex, expected) {
    var matched = string.match(regex);
    var result = matched !== null && matched[0] === string;
    if(result !== expected) {
        if(result) {
            console.log("Error: '" + string + "' is " + regex_name);
        } else {
            console.log("Error: '" + string + "' is not " + regex_name);
        }
    }
    return result === expected;
}

// expected_subgroups_matches should only be the subgroups
function test_all_matches(string, regex_name, regex, expected_subgroups_matches) {
    function print_error() {
        console.log("Error: '" + string + "does not match all subgroups");
        console.log("    Matched subgroups  " + JSON.stringify(matched.splice(1)));
        console.log("    Expected subgroups " + JSON.stringify(expected_subgroups_matches));
    }
    if(!test_is_full_match(string, regex_name, regex, true)) {
        return false;
    }
    var matched = string.match(regex);
    if(matched.length != expected_subgroups_matches.length + 1) {
        print_error();
        return false;
    }
    for(var i = 1; i < matched.length; i++) {
        if(matched[i] != expected_subgroups_matches[i-1]) {
            print_error();
            return false;
        }
    }
    return true;
}
